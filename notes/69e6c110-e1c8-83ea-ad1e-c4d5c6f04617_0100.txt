What you’ve got there is essentially a modernized, reality-checked checklist for building a clean ffmpeg X11 screen capture command—specifically tuned to avoid outdated advice and flags that sound useful but actually degrade behavior.

I’ll walk through it in plain terms, but also clarify why these matter in real pipelines (OBS, wf-recorder evolution, ffmpeg internals, etc.).
1) Removing “low latency” flags

Key idea: Not all latency flags apply to all inputs.

    -fflags nobuffer and -flags low_delay are mainly useful for:

        network streams (RTSP, UDP)

        demuxers with buffering stages

Problem with x11grab:

    It’s not a buffered stream—it’s a live framebuffer pull

    These flags can:

        disrupt frame pacing

        introduce jitter

        cause inconsistent timestamps

👉 So removing them is correct. They’re misapplied, not just unnecessary.
2) Skipping probing flags

    -probesize and -analyzeduration help ffmpeg figure out unknown streams

x11grab reality:

    Format is already known and raw

    No codecs to detect

    No stream ambiguity

👉 These flags literally do nothing here except clutter and confuse.
3) Explicit capture origin (+0,0)

This is more important than it looks.

    ${DISPLAY} alone can be ambiguous on:

        multi-monitor setups

        non-zero origin layouts (common with Xinerama/XRandR)

Adding:

-i "${DISPLAY}+0,0"

👉 Guarantees:

    Capture starts at the top-left of the virtual framebuffer

    Prevents offset bugs (shifted or partial capture)

4) Explicit framerate

-framerate ${FPS}

This is critical for input timing, not encoding.

    x11grab doesn’t “push” frames—it’s polled

    Without -framerate, timing becomes:

        inconsistent

        dependent on system load

👉 This ensures stable capture pacing.
5) Avoid forcing wallclock timestamps

-use_wallclock_as_timestamps 1

This used to be a workaround for broken timing.

Today:

    x11grab timestamping is already stable

    Forcing wallclock can:

        desync A/V (if audio added later)

        introduce drift

👉 Only use it if you actually observe jitter.
6) Thread utilization

THREADS="$(nproc)"

    libx264 scales well with threads

    Hardcoding -threads 2 is outdated unless:

        you’re intentionally limiting CPU

👉 Modern CPUs = leave performance on the table otherwise.
7) GOP (keyframe interval)

-g $((FPS * 2))

This is one of the most important additions.

    Defines how often keyframes occur

    Example:

        30 FPS → GOP = 60 → keyframe every 2 seconds

Why it matters:

    Better seeking

    Better compression

    Cleaner editing cuts

👉 Without this, ffmpeg may choose suboptimal intervals.
8) Thread queue buffering

-thread_queue_size 1024

    Buffers frames between capture and encoding threads

Why needed:

    Prevents frame drops when:

        CPU spikes

        disk I/O stalls

👉 This is very practical, not theoretical.
9) Preset + zerolatency tuning

-preset veryfast -tune zerolatency

    preset: speed vs compression tradeoff

    zerolatency:

        disables lookahead

        reduces encoder buffering

👉 Good for:

    live capture

    low-lag workflows

Tradeoff:

    Slightly larger files

10) Pixel format

-pix_fmt yuv420p

This is about compatibility, not quality.

    Required by:

        browsers

        many players

        editing tools

👉 Without it:

    You risk playback issues (especially with RGB or yuv444p)

11) Geometry via xwininfo

Using:

xwininfo -root

to get width/height ensures:

    exact framebuffer size

    no guesswork

👉 This avoids:

    cropping

    black borders

    mismatched resolutions

12) Multi-monitor behavior

Important conceptual point:

    Root window = entire virtual desktop

    Not a single monitor

👉 If you want one monitor:

    you must switch to:

        xrandr region selection

        or manual offsets

13) Deterministic filenames

Timestamp-based naming:

capture-YYYYMMDD-HHMMSS.mp4

👉 Prevents:

    accidental overwrites

    ambiguity in logs/workflows

14) Faststart

-movflags +faststart

    Moves MP4 metadata (moov atom) to the front

👉 Result:

    video starts playing immediately when opened or streamed

15) Clean final command philosophy

This is the big-picture principle:

    Only include flags that are proven to affect this pipeline

Why this matters:

    ffmpeg has thousands of options

    many guides copy-paste irrelevant flags

    mixing unrelated flags causes:

        undefined behavior

        subtle bugs

        harder debugging

What this all boils down to

This checklist is correcting a very common problem:

    Cargo-cult ffmpeg commands

Instead, it enforces:

    input-specific correctness (x11grab ≠ network stream)

    modern encoder behavior

    predictable timing

    real-world stability under load

A clean example command (result of all steps)

Putting it all together:

ffmpeg \
  -f x11grab \
  -framerate "$FPS" \
  -video_size "${WIDTH}x${HEIGHT}" \
  -thread_queue_size 1024 \
  -i "${DISPLAY}+0,0" \
  -c:v libx264 \
  -preset veryfast \
  -tune zerolatency \
  -threads "$(nproc)" \
  -g $((FPS * 2)) \
  -pix_fmt yuv420p \
  -movflags +faststart \
  "capture-$(date +%Y%m%d-%H%M%S).mp4"
